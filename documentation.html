System Overview

1. System Overview

Bugify is a lightweight, browser-based bug tracking system built for the QA team at Vantage Circle. It allows QA testers to log bugs during testing sessions and gives team leads a live dashboard to monitor, filter, and export all reported bugs — with no app installation or account required.

🧑‍💻
QA Testers
Submit bugs via a simple form during sprint testing sessions
📊
Team Leads
View all bugs, apply filters, track status, and export reports
🛠
Developers
Receive bug assignments and update status directly in Google Sheets
✅
Zero setup for testers

QA testers just open a browser link. No login, no app install, no account needed. The team lead manages access to the dashboard separately.

2. Technology Stack

Bugify uses only widely available, free tools — no paid services, no npm packages, no build process.

🌐
Vanilla HTML / CSS / JavaScript
The entire frontend is written in plain web standards — no frameworks (React, Vue, etc.), no dependencies. Works in any modern browser.
Frontend
⚙️
Google Apps Script
The serverless backend. Deployed as a Web App, it handles all API requests — receiving bug submissions (POST) and serving dashboard data (GET).
Backend / API
📊
Google Sheets
The database. Bugs are stored in a "Master Bug Sheet" and also in individual tester sheets. Team leads can also view/update data directly in Sheets.
Database
🚀
GitHub Pages
Hosts the static HTML files. Every push to the main branch automatically redeploys via GitHub Actions.
Hosting

4. Key Features

📋
Multi-Bug Session Submission

Testers can log multiple bugs in one session and submit them all at once as a batch. Session details (tester name, sprint, product) apply to all bugs in the session.

🆔
Auto-Generated Bug IDs

Each bug gets a unique, sequential ID like KUM-0001 (3-letter tester prefix + 4-digit number). IDs never repeat, even if two bugs are submitted simultaneously.

🔍
Live Dashboard & Filters

Team leads can filter bugs by sprint, product, tester, severity, status, and date range. All filters work together. Stat cards update instantly when filters change.

📤
CSV Export

Export the currently filtered view to a CSV file with one click. The filename automatically includes the tester name and date for easy filing.

🃏
Table & Card Views

Switch between a structured data table and a card-based kanban-style grid. Both views respect active filters and show the same data.

⚡
Smart Caching

The dashboard caches data locally (30-minute freshness window). On reload, saved data shows instantly — no blank screen or spinner on every visit.

1
Fill in Session Details
At the top, select your Name, enter the Sprint number (e.g. Sprint 42), and select the Product you are testing. These three fields apply to every bug you add in this session. They are saved in your browser so you don't need to re-enter them next time.
2
Click "+ Add Bug"
Press the blue + Add Bug button. A bug card will expand below the session details. Fill in the bug's details (see Section 6 for all fields). You can add as many bugs as you find in one session — click the button again for each new bug.
3
Fill in Bug Details
Each bug card has: Module, Platform, Description, Expected Result, Actual Result, Severity, Proof Link, and Assigned Developer. Fields marked with a red asterisk (*) are required. The form highlights missing fields in red if you try to submit without them.
4
Submit All Bugs at Once
Once all bug cards are filled, click the green Submit All Bugs button at the bottom. All bugs are sent together as a single batch. A confirmation screen will appear showing "Bugs Submitted!" along with a summary of your session. The form then resets for the next session.
⚠️
Proof Link is required

Always attach a screenshot or screen recording. Upload it to Google Drive, Confluence, or any shareable link and paste the URL in the Proof Link field. Bugs without proof are harder to reproduce and investigate.

6. Bug Form Fields Explained

Field Required What to Write Example
Module Required The feature area or section of the app where the bug was found. Rewards Catalogue, Dashboard, Login, Notifications
Platform Required The device or access method used during testing. Web, Android, iOS, API, Desktop
Description Required A short, clear summary of what went wrong. Be specific — include page name, action taken, and observed issue. "Redeeming a reward on the Catalogue page shows a success toast but the order is not created"
Expected Result Required What should happen when following the correct flow. "Order should appear in My Orders within 30 seconds"
Actual Result Required What actually happened instead. The difference between Expected and Actual is the bug. "Toast shows 'Order placed' but My Orders shows nothing"
Severity Required How severely the bug impacts the user. Select one from the severity scale (see below). Critical
Proof Link Required URL to a screenshot, video, or recording showing the bug. Upload to Google Drive or similar, set to "Anyone with link can view", then paste the URL here. drive.google.com/file/d/…
Assigned Dev Optional The developer or team member responsible for fixing this bug. Can be left blank if unknown. Abhishek Das, Gaurav, Rahul Roy
Status Auto Automatically set to Open when submitted. Updated later by the team via Google Sheets or the dashboard. Open

Severity Scale

LevelWhen to useDefinition
🔴 Critical App is broken / data loss The entire feature or product is non-functional. Users cannot complete a core task. Requires immediate fix before release.
🟠 High Major feature broken A major feature is broken with no workaround. Affects a large number of users or a critical business flow.
🟡 Medium Feature broken, has workaround A feature is broken but a workaround exists. Should be fixed but does not block the release.
🟢 Low Minor issue Minor cosmetic or functional issue with minimal impact on users. Can be addressed in a future sprint.
⚪ Trivial Cosmetic / nitpick Typos, alignment issues, minor visual inconsistencies. Does not affect functionality.

Filters, Views & Export

8. Filters & Search

🔎
Full-Text Search

Type anything in the search box to search across Bug ID, description, tester name, module, product, sprint, assigned developer, and platform simultaneously. Results update as you type (400ms debounce).

📅
Date Range Filter

Select a From and To date to narrow down bugs submitted in a specific time window. A summary (e.g. "From 2026-03-01 → 14 bugs") appears automatically. Click ✕ Clear dates to reset.

🏷
Sprint / Product / Severity / Status

Four dropdowns let you drill down to exactly the subset of bugs you need. All dropdowns and tester pills work together — every active filter stacks on top of the others.

🔃
Column Sorting

Click any column header in the table to sort by that column. Click again to reverse the sort direction. A ▲ or ▼ indicator shows the active sort. Defaults to newest first (Submitted date descending).

Table View vs Card View

☰ Table View

All bugs in a structured spreadsheet layout. Best for scanning many bugs quickly, sorting by column, and seeing all data at once. Supports 50 bugs per page with pagination.

Use when: reviewing all bugs in a sprint, preparing a status report.

⊞ Card View

Each bug displayed as a compact card showing ID, title, severity badge, status badge, tester name, and proof link. Great for a visual overview by tester or severity.

Use when: doing a quick triage, showing bugs in a standup or review.

📤 Exporting to CSV

1
Apply your filters
Set any filters you need — e.g. select Sprint 42 + tester "Kumar Ankit" + Status "Open". The export always exports exactly what is currently visible after filtering.
2
Click "↓ Export CSV"
The green Export CSV button is in the top-right of the toolbar. Clicking it downloads a .csv file to your computer. The filename automatically includes the tester name and date (e.g. bugs_Kumar_Ankit_2026-03-25.csv).
3
Open in Excel / Google Sheets
The CSV contains all 14 columns: Bug ID, Tester, Sprint, Product, Module, Platform, Description, Expected, Actual, Severity, Proof Link, Status, Assigned To, Submitted At.
🔄
Refreshing Data

The dashboard caches data for 30 minutes. Click ↻ Refresh at any time to force-fetch the latest bugs from Google Sheets. If a new bug was just submitted and doesn't appear, refresh.

Bug ID Format & Benefits

11. Bug ID Format

KUM-0017
Example Bug ID
KUM
Prefix
-
0017
Sequence
Prefix = first 3 letters of the tester's name, uppercase (KUM from "Kumar Ankit").
Sequence = 4-digit zero-padded number, increments per tester, never resets.
IDs are generated by Google Apps Script using persistent script properties, so they are collision-safe even with simultaneous submissions.
💡
Referencing bugs in conversations

Always refer to bugs by their ID in Slack, JIRA, or emails — e.g. "KUM-0017 is blocking the release". This ensures everyone is talking about the same bug and it can be found instantly in the dashboard using the search box.

Setup & Deployment

13. Setup & Deployment

Initial setup is a one-time task for the system administrator. After setup, the system runs automatically with zero maintenance.

Backend — Google Apps Script

1
Create a Google Sheet
Go to sheets.google.com → New Sheet. Copy the Sheet ID from the URL (the long string between /d/ and /edit). Paste it into google-apps-script.js as the value of SHEET_ID.
2
Deploy the Web App
Open script.google.com → New Project. Paste the contents of google-apps-script.js. Click Deploy → New Deployment → Web App. Set "Execute as: Me", "Who has access: Anyone". Copy the deployment URL.
3
Run one-time setup functions
In the Apps Script editor, run setupTrigger() to install the auto-sync onEdit trigger. Run setupValidation() to add Status dropdowns to all tester sheets.
4
Update config.js
Paste the Web App URL into config.js as scriptUrl. Set dashboardPassword to a secure password. Commit and push — GitHub Actions will deploy to GitHub Pages automatically.
⚠️
Re-deploying after backend changes

If you edit google-apps-script.js, you must manually create a new deployment in the Apps Script editor (Deploy → Manage Deployments → New Version). The URL stays the same — the change takes effect immediately after the new deployment is saved.

Frequently Asked Questions

15. Frequently Asked Questions

Q
I submitted a bug but it doesn't appear on the dashboard. What do I do?
Click the ↻ Refresh button on the dashboard to bypass the 30-minute cache and fetch the latest data. If it still doesn't appear after refreshing, check whether the Google Apps Script deployment is active and that the Sheet ID in the script matches the actual spreadsheet.
Q
Can I edit or delete a bug after submitting?
The form does not support editing submitted bugs. To correct a bug, open the Google Sheet directly (ask your team lead for access) and edit the row manually. You can also submit a new bug with the corrected information and note in the description that it supersedes the previous one.
Q
How do I update the status of a bug?
Open the Google Spreadsheet and find your tester sheet (tab named after you). In column K (Status), click the cell for the bug you want to update — a dropdown will appear. Select the new status. The change automatically syncs to the Master Bug Sheet within seconds and will appear on the dashboard on next refresh.
Q
My name is not in the tester dropdown. What should I do?
Contact your team lead or system admin and ask them to add your name to the TESTERS array in index.html. After the change is pushed to GitHub, your name will appear in the dropdown.
Q
The form shows an error "Submission failed — check your connection". What's wrong?
This means the request to Google Apps Script timed out (after 25 seconds) or failed. Possible causes: (1) no internet connection, (2) the Apps Script deployment was revoked or expired and needs to be redeployed, (3) Google's servers are temporarily slow. Try again in a few minutes. If it persists, the admin should check the deployment status.
Q
How do I share proof (screenshots/videos)?
Upload to Google Drive → right-click the file → Share → "Anyone with the link can view" → Copy link. Paste that link in the Proof Link field. Alternatively use Confluence, Loom, or any other platform that gives a public or shared view URL. Avoid attaching files directly — use shareable links only.
